# Environment Sensor Firmware

CircuitPython firmware for the **Arduino Nano RP2040 Connect** that reads
temperature and humidity from a DHT20 sensor, displays both values on an
SSD1306 OLED, and publishes readings to Home Assistant via MQTT Discovery.

OTA firmware updates are served by the
[nano_backbone](../README.md) server.

---

## Hardware

| Component | Notes |
|---|---|
| Arduino Nano RP2040 Connect | Main board |
| SSD1306 OLED display | 128×64 px, I²C, 3.3 V |
| DHT20 temperature/humidity sensor | I²C, address `0x38`, 3.3 V |

### Wiring

Both the SSD1306 and DHT20 share the same I²C bus.

| Sensor pin | Nano RP2040 Connect pin |
|---|---|
| SDA (both devices) | `SDA` (D4) |
| SCL (both devices) | `SCL` (D5) |
| VCC (both devices) | `3.3V` |
| GND (both devices) | `GND` |

> **I²C addresses**: SSD1306 defaults to `0x3C` (try `0x3D` if the display
> does not initialise). DHT20 is always `0x38`.

## Firmware installation

Copy the following files from this directory to the root of the `CIRCUITPY`
drive:

```
boot.py
code.py
ota.py
captive.py
sensor.py
display.py
mqtt_ha.py
```

Also copy `font5x8.bin` to the root of `CIRCUITPY`. This font file is required
by `adafruit_framebuf` for text rendering on the OLED display. Download it from
the `examples/` folder of the
[Adafruit CircuitPython framebuf repository](https://github.com/adafruit/Adafruit_CircuitPython_framebuf/blob/main/examples/font5x8.bin).

Do **not** copy `settings.toml` — it is generated by the captive portal on
first boot.

## First-boot provisioning

On first boot (no `settings.toml`), the device starts a Wi-Fi access point
named **`env-sensor-setup`**.

1. Connect your phone or laptop to `env-sensor-setup` (no password).
2. Open `http://192.168.4.1/` in a browser.
3. Fill in the form:

| Field | Description |
|---|---|
| **WiFi SSID** | Your Wi-Fi network name |
| **WiFi Password** | Your Wi-Fi password (leave blank for open networks) |
| **Server URL** | nano_backbone server, e.g. `http://192.168.1.100:8000` |
| **API Key** | Device API key from `POST /api/v1/devices/register/` |
| **MQTT Broker** | Hostname or IP of your MQTT broker |
| **MQTT Port** | Broker port (default `1883`) |
| **MQTT Username** | Optional — leave blank for anonymous access |
| **MQTT Password** | Optional — leave blank for anonymous access |
| **Device name** | Human-readable label shown in Home Assistant, e.g. `Living Room Sensor` |
| **Poll interval** | Seconds between sensor readings (default `30`, minimum `2`) |

4. Click **Save and Reboot**. The device reboots and connects to your network.

### Registering the device

Before provisioning, register the device with the nano_backbone server to
obtain an API key:

```bash
curl -X POST http://<server>:8000/api/v1/devices/register/ \
  -H "Content-Type: application/json" \
  -d '{"name": "living-room-sensor", "device_type": "arduino_nano_rp2040_connect"}'
```

Save the returned `api_key` — it is shown only once.

## Home Assistant integration

After the first successful boot the device publishes MQTT Discovery config
payloads. Home Assistant picks them up automatically and creates:

- **`<Device name> Temperature`** — °C, device class `temperature`
- **`<Device name> Humidity`** — %, device class `humidity`

Both entities are grouped under a single HA device identified by the board's
hardware chip ID (`DEVICE_ID`), which is stable across reboots and OTA updates.

No manual HA configuration is required beyond having the MQTT integration
enabled.

## OTA firmware updates

### Zip layout

The nano_backbone server expects a zip archive containing the firmware files
at the **root level** (no subdirectory). `boot.py` is intentionally excluded
— it is the OTA safety boundary and must not be overwritten by an update.

`build_zip.sh` automatically generates a `firmware_manifest.txt` and includes
it in the zip. This manifest tells `boot.py` which files to restore during a
rollback, so `boot.py` never needs to be modified when new files are added to
the firmware.

```
env_sensor_<version>.zip
  code.py
  ota.py
  captive.py
  sensor.py
  display.py
  mqtt_ha.py
  firmware_manifest.txt
  lib/some_library.mpy    ← only when bundling a new library dependency
```

### Bundling new library dependencies

If a release introduces a library that is not yet installed on target devices,
stage it in `lib/` before building so it gets included in the zip and deployed
alongside the firmware files:

```bash
# Example: first release to require neopixel.mpy
cp /path/to/circuitpython-bundle/lib/neopixel.mpy example_firmware/lib/
```

The `lib/` directory is gitignored. Only stage files needed for the current
release — once a library is on all devices it no longer needs to be bundled.

### Building the zip

```bash
cd example_firmware
./build_zip.sh 1.0.0
```

Output:

```
Created : env_sensor_1.0.0.zip
SHA256  : <hex digest>

Upload this zip via the Django admin (SHA-256 is computed automatically on save):
  http://<server>:8000/admin/firmware/firmwarerelease/add/
```

### Uploading to the server

Use the Django admin at `http://<server>:8000/admin/` to create a new
**Firmware Release**:

1. Log in to the admin.
2. Go to **Firmware → Firmware releases → Add**.
3. Set **Version** to e.g. `1.0.0`.
4. Set **Device type** to `arduino_nano_rp2040_connect`.
5. Upload the zip produced by `build_zip.sh`.
6. Check **Is latest** to make it the active release.
7. Save — the SHA-256 is computed automatically.

On the next boot all registered devices of this type will receive and apply
the update.

## Integration test checklist

These steps require the physical device:

- [ ] Copy firmware files to `CIRCUITPY` and reboot.
- [ ] Connect to `env-sensor-setup` AP and complete the provisioning form.
- [ ] Confirm the OLED shows temperature and humidity after reboot.
- [ ] Confirm two sensor entities appear in Home Assistant automatically.
- [ ] Confirm entity names match the **Device name** entered during provisioning.
- [ ] Upload a new firmware release to the nano_backbone server (version > current).
- [ ] Reboot the device and confirm it downloads and applies the update.
- [ ] Confirm the server logs an `update_success` event for the device.
- [ ] Power-cycle the device and confirm it boots into the updated firmware.
